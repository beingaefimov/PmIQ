# PM IQ Standalone Agent Sample

Этот пример демонстрирует, как использовать слой PM IQ (Universal Edition) с локальной LLM без зависимости от GitHub Copilot CLI.

## Архитектура

1. **LLM Server**: `llama.cpp` запускает локальную модель (например, Llama-3-8B-Instruct) и предоставляет OpenAI-совместимый API.
2. **Agent (Python)**: Реализует паттерн ReAct (Reasoning and Acting). Он решает, какой инструмент вызвать, парсит ответ и формирует финальный вывод.
3. **MCP Layer**: Python-скрипт подключается к MCP-серверу PM IQ через `stdio` и вызывает инструменты (Jira, SAP, Primavera и т.д).

### Запуск llama.cpp server

Пробросить сеть Windows-WSL если используется WSL

В PowerShell:

```powershell
notepad $env:USERPROFILE\.wslconfig
```

Добавить содержимое:

```bash
[wsl2]
localhostForwarding=true
networkingMode=mirrored
```

Сохранить файл.

Перезапустить WSL:

```powershell
wsl --shutdown
```

Затем открыть WSL заново.

Запуск llama.cpp server

```bash
./llama-server -m "models/llama-3-8b-instruct.gguf" -c 32000 --host 0.0.0.0 --port 8080
```

*Убедитесь, что сервер отвечает на `http://localhost:8080/v1/models`.*

Но! По умолчанию температура может быть установлена на 0.8, что отлично для чат-ботов, но катастрофично для агентов.
Рекомендуемая команда запуска для агентных задач:

```bash
./llama-server -m "models/ваша_модель.gguf" ^
  --host 0.0.0.0 ^
  --port 8080 ^
  --ctx-size 32000 ^
  --temp 0.1 ^
  --top-p 0.9 ^
  --repeat-penalty 1.15 ^
  --mirostat 2 ^
  --mirostat-lr 0.1 ^
  --mirostat-ent 5.0
```

Где:
--temp 0.1: Делает выбор токенов почти детерминированным. Модель будет выбирать наиболее вероятный следующий токен, что критично для генерации валидного JSON и соблюдения структуры Thought -> Action.
--top-p 0.9: (Nucleus sampling) Ограничивает выборку только наиболее вероятными токенами, отсекая "хвост" распределения, где живут галлюцинации.
--repeat-penalty 1.15: Самый важный параметр для ReAct -локальные модели могут зацикливаться (например, 10 раз подряд выдавать Action: get_project_schedule). Штраф за повторение ломает этот цикл.
--mirostat 2: Алгоритм, который динамически регулирует температуру, чтобы поддерживать "перплексию" (непредсказуемость) текста на стабильном уровне. Для локальных 7B-8B моделей это часто дает более связные и менее "бредовые" рассуждения, чем фиксированная температура.

Тем не менее для мало-B моделей, неизбежным будет эффект: если модель "решила", что "Brigade_1" (из мок-данных) звучит некрасиво, и ее внутренние веса (и примеры промптов) подсказывают, что после слова "Dev" чаще идет имя "Sarah", она может сделать замену Brigade_1 в Sarah (Dev) даже при температуре 0.1

## Как это работает (Поток ReAct)

1. Пользователь задает вопрос: *"Каков статус критического пути?"*
2. Агент формирует промпт со списком выбранных инструментов из PM IQ.
3. LLM генерирует:
   ```text
   Thought: Мне нужно получить расписание проекта.
   Action: get_project_schedule
   Action Input: {"project_name": "Миграция ERP"}
   ```
4. Python-скрипт перехватывает это, вызывает реальный MCP-инструмент `get_project_schedule`.
5. MCP возвращает JSON с данными из Primavera/Jira.
6. Python добавляет это как `Observation` в историю чата.
7. LLM анализирует данные и выдает `Final Answer` на естественном языке.

## Настройка

В файле `pm_iq_agent.py` измените:
- `LLM_BASE_URL`, если `llama.cpp` запущен на другом порту или хосте.
- Словарь `env` в `StdioServerParameters`, добавив реальные API-ключи.

## Замечание о «Меню виджетов»

«Меню виджетов» (Widget Menu) - это **ключевой архитектурный мост** между декларативными инструкциями в `SKILL.md` и программной генерацией сложного JSON на стороне MCP сервера.

Это механизм, который говорит серверу: *«Когда вызывается этот инструмент, заранее приготовь вот эти конкретные диаграммы, чтобы LLM могла просто выбрать подходящую, а не пытаться нарисовать её сама»*.

---

### Какую проблему это решает?

LLM 7B–8B **катастрофически плохо** генерируют сложные, строго типизированные JSON-структуры (такие как конфигурация ECharts) с нуля. Они гарантированно:
- Допустят синтаксическую ошибку (забудут запятую или скобку).
- «Сгаллюцинируют» несуществующие поля.
- Искажают данные, пытаясь их «улучшить».

**Решение:** Перенести генерацию строгого JSON на сторону Python-сервера (где это делается детерминировано и без ошибок), а LLM оставили роль **куратора/выборщика**.

### ЖЦ «Меню виджетов»

#### Этап А: Декларация (в SKILL.md)

Автор плагина в YAML-фронтматтере скилла объявляет, какие типы визуализаций поддерживаются в контексте именного этого скилла, зачем они нужны и как сопоставить данные CSV с осями диаграммы:

```yaml
available_widgets:
  - type: BarChart
    description: "Гистограмма индивидуальной загрузки. Используйте, если нужно выявить конкретных перегруженных сотрудников."
    data_mapping:
      x_axis: "resource_name"
      y_axis: "allocation_pct"
      
  - type: ActionCard
    description: "Интерактивная карточка для создания плана реагирования. Используйте, если есть активные риски с высоким воздействием."
    data_mapping:
      filter: "impact=High AND status=Active"
      id_field: "risk_id"
      description_field: "description"
```

**Смысл:** `data_mapping` выступает мостом между универсальным генератором и специфической структурой CSV-файла данного скилла.
В одном скилле модно задекларировать несколько диаграмм одного типа но разного назначения в контексте этого скилла.

#### Этап Б: Сборка меню

При старте сервера функция `build_dynamic_widget_menu` сканирует все `SKILL.md`. Для каждого объявленного виджета она:

1. Берёт `type` (например, `BarChart`) и находит соответствующую универсальную Python-функцию в словаре `WIDGET_GENERATORS` (например, `generate_echarts_barchart`).
2. Связывает эту функцию с `description` и `data_mapping` из YAML.

**Результат:** Словарь, где ключом является имя инструмента (например, `get_resource_histogram`), а значением — список готовых конфигураций:

```python
[
    {
        "type": "BarChart",
        "description": "...",
        "data_mapping": {"x_axis": "resource_name", "y_axis": "allocation_pct"},
        "func": <function>
    },
    ...
]
```

#### Этап В: Исполнение (в call_tool)

Когда LLM вызывает инструмент (например, `get_resource_histogram`), сервер:

1. Загружает и фильтрует данные из соответствующего сервиса.
2. Берёт конфигурации виджетов из "Меню" для этого инструмента.
3. Автоматически вызывает универсальные функции, передавая им и данные, и маппинг:
   ```python
   widget_data = config["func"](filtered_data, config["data_mapping"])
   ```
4. Универсальная функция строит JSON диаграммы, используя колонки, указанные в `data_mapping`.
5. Сервер добавляет поле `"description"` в каждый сгенерированный JSON и формирует итоговый массив `widgets`.

#### Этап Г: Выбор и рендеринг LLM

LLM получает `Observation`, содержащий сырые данные (`data`) и массив готовых JSON-шаблонов (`widgets`).

1. **Анализ:** LLM читает поле `description` каждого виджета и сопоставляет его с намерением пользователя.

2. **Выбор:** Если запрос пользователя совпадает с описанием (например, пользователь спрашивает про "перегрузку сотрудников", а в описании BarChart написано "выявить конкретных перегруженных сотрудников"), LLM выбирает этот виджет. Может быть выбрано несколько виджетов одновременно.

3. **Контекстуализация:** LLM обязана заменить технический плейсхолдер в поле `"title"` (например, `"title": "BarChart"`) на осмысленный, контекстно-зависимый заголовок (например, `"title": "Загрузка команды разработки проекта ERP"`).

4. **Гарантия целостности:** LLM НЕ ДОЛЖНА изменять структуру `option`, `series`, `data` или любые другие поля, сгенерированные сервером. Она копирует JSON целиком.

5. **Вывод:** LLM вставляет выбранные и обновлённые JSON-блоки в свой `Final Answer`, оформляя их как стандартные блоки кода ```json ... ```, которые затем парсятся и отрисовываются UI-клиентом.
