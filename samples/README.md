# PM IQ Standalone Agent Sample

- Внимание! Это не продуктовый код! Только для тестов слоя локально, одним пользователем.
- This is not production code! For local layer testing only, by a single user.

Пример демонстрирует, как использовать слой PM IQ (Universal Edition) с локальной LLM.

Агент в примере поддерживает **два режима работы ReAct-цикла**, которые можно переключать в UI:

- **Концентратор + ReACT (v0.2)** - подход с концентрацией знаний + классический ReAct-цикл с накапливаемой историей.
- **Stateless ReACT (v0.01)** - цикл «с чистого листа». На каждой итерации LLM получает полный текст выбранных роутером методологий (и предформатированный каталог всех виджетов из них, дублирующий описания виджетов в методологиях) + растущий блок «По данным компании известно, что: …» + исходный вопрос. История ReAct не накапливается - требует бОльшего контекстного окна LLM

## Архитектура

1. **LLM Server**: `llama.cpp` запускает локальную модель (например, T-lite-it-2.1) и предоставляет OpenAI-совместимый API.
2. **Agent (Python)**: Реализует паттерн ReAct (Reasoning and Acting). Он решает, какой инструмент вызвать, парсит ответ и формирует финальный вывод.
3. **MCP Layer**: Python-скрипт подключается к MCP-серверу PM IQ через `stdio` и вызывает инструменты (моки как будто из Jira, SAP, Primavera и т.д).
4. **FastAPI + SSE**: `pm_iq_api.py` оборачивает агента в HTTP-сервер с SSE-стримингом логов в реальном времени

### Запуск llama.cpp server

Пробросить сеть Windows-WSL если используется WSL

В PowerShell:

```powershell
notepad $env:USERPROFILE\.wslconfig
```

Добавить:

```bash
[wsl2]
localhostForwarding=true
networkingMode=mirrored
```

Сохранить файл.

Перезапустить WSL:

```powershell
wsl --shutdown
```

Затем открыть WSL заново.

Запуск llama.cpp server

```bash
./llama-server -m "models/T-lite-it-2.1-Q4_K_M.gguf" -c 0 --host 0.0.0.0 --port 8080
```

*Убедитесь, что сервер отвечает на `http://localhost:8080/v1/models`.*

Но! По умолчанию температура может быть установлена на 0.8, что отлично для чат-ботов, но катастрофично для агентов.
Рекомендуемая команда запуска для агентных задач:

```bash
./llama-server -m "models/T-lite-it-2.1-Q4_K_M.gguf" ^
  --host 0.0.0.0 ^
  --port 8080 ^
  --ctx-size 0 ^
  --temp 0.1 ^
  --top-p 0.9 ^
  --repeat-penalty 1.15 ^
  --mirostat 2 ^
  --mirostat-lr 0.1 ^
  --mirostat-ent 5.0
```

Где:
- --temp 0.1: Делает выбор токенов почти детерминированным. Модель будет выбирать наиболее вероятный следующий токен, что критично для генерации валидного JSON и соблюдения структуры Thought -> Action.
-  --top-p 0.9: (Nucleus sampling) Ограничивает выборку только наиболее вероятными токенами, отсекая "хвост" распределения, где живут галлюцинации.
-  --repeat-penalty 1.15: Самый важный параметр для ReAct -локальные модели могут зацикливаться (например, 10 раз подряд выдавать Action: get_project_schedule). Штраф за повторение ломает этот цикл.
-  --mirostat 2: Алгоритм, который динамически регулирует температуру, чтобы поддерживать "перплексию" (непредсказуемость) текста на стабильном уровне. Для локальных 7B-8B моделей это часто дает более связные и менее "бредовые" рассуждения, чем фиксированная температура.

Тем не менее для мало-B моделей, неизбежным будет эффект: если модель "решила", что "Brigade_1" (из мок-данных) звучит некрасиво, и ее внутренние веса (и примеры промптов) подсказывают, что после слова "Dev" чаще идет имя "Sarah", она может сделать замену Brigade_1 в Sarah (Dev) даже при температуре 0.1

## Режимы работы

Агент поддерживает два режима, которые можно выбирать через UI или передавать в API per-request

### 1. Концентратор + ReACT (`concentrator`)

Классический подход с двумя LLM-вызовами в пайплайне:

1. **Маршрутизация** - LLM выбирает 1–3 релевантных плагина по вопросу пользователя.
2. **Концентрация знаний** - второй LLM-вызов сжимает полные методички выбранных скиллов в 3–10 раз, оставляя только релевантное вопросу. На выходе - концентрированный контекст + запланированные инструменты + запланированные виджеты + пошаговый план.
3. **ReAct-цикл с историей** - модель работает в классическом цикле `Thought → Action → Observation → ... → Final Answer`, история диалога накапливается.
4. **Постобработка** - системные виджеты (`BarChart`, `LineChart`, `ScatterChart`) рендерятся из кэша данных через плейсхолдеры; модельные (`action_card`) генерируются LLM в Final Answer.

Анти-зацикливание: после 2 повторов одного инструмента с теми же аргументами - принудительный Final Answer

### 2. Stateless ReACT (`stateless`)

Цикл «с чистого листа» на каждой итерации. Для притягивания актуального контекста - обновления применяемых знаний по фактической обстановке на шаге.

1. **Маршрутизация заново** - LLM выбирает плагины, учитывая накопленные к этому моменту данные.
2. **Промпт с чистого листа** - модель получает полные методологии отобранных плагинов (без сжатия) + накопленный блок «По данным компании известно, что: …» + исходный вопрос. История ReAct не накапливается.
3. **План + первый шаг** - модель составляет пронумерованный план и выполняет первый шаг (вызов инструмента или Final Answer, если план одношаговый).
4. **Накопление данных** - результат вызова инструмента приписывается к блоку фактических данных с пометкой «получены вызовом `<tool>` с аргументами `<args>`» (модель видит историю вызовов, но не видит ReAct-историю).
5. **Цикл повторяется** - пока модель не выдаст одношаговый план `Final Answer`.

**Правило формирования виджетов (критично):** модель может собирать данные на любом шаге плана, но формировать структуру виджетов - только на последнем шаге (в финальном ответе). Каталог виджетов приведён в промпте «для сведения», чтобы модель понимала, какие данные нужно собрать.

**Анти-loop** через стек последних 2 вызовов: если модель первым шагом плана предлагает подряд-дубликат (тот же инструмент + те же args, что в вершине стека), система игнорирует его и выполняет следующий не-дубликат шаг из плана. Не-подрядные повторы не пропускаются, но модель видит их в блоке данных с припиской и сама должна избегать.

**Системная защита виджетов**: перед возвратом Final Answer проверяется, что для каждого плейсхолдера виджета есть данные питающего инструмента

## Настройка

В файле `pm_iq_agent.py` изменить:

- `LLM_BASE_URL` - если `llama.cpp` запущен на другом порту или хосте (по умолчанию `http://localhost:3101/v1`).
- `LLM_MODEL_NAME` - имя модели (по умолчанию `local-model`).
- `LOOP_MODE` - режим по умолчанию (`"concentrator"` или `"stateless"`). Может быть переопределён per-request через параметр `mode` метода `run(query, mode=...)` или через поле `mode` в API-запросе `/api/ask`.
- Словарь `env` в `StdioServerParameters`, добавив реальные API-ключи.

В файле `pm_iq_api.py`:

- `DEFAULT_MODE` - режим по умолчанию для API-запросов (по умолчанию `"concentrator"`).
- `ALLOWED_MODES` - список допустимых значений режима (по умолчанию `("concentrator", "stateless")`).

## Замечание о виджетах

«Виджеты» - это **ключевой архитектурный мост** между декларативными инструкциями в `SKILL.md` и программной генерацией сложного JSON на стороне Python-сервера.

Это механизм, который говорит серверу: *«Когда LLM в Final Answer захочет показать диаграмму X - есть готовый детерминированный построитель, который соберёт её из данных конкретного инструмента, а LLM остаётся только выбрать подходящий intent и указать фильтр»*.

---

### Какую проблему это решает?

LLM 7B–8B **катастрофически плохо** генерируют сложные, строго типизированные JSON-структуры (такие как конфигурация ECharts) с нуля. Они гарантированно:
- Допустят синтаксическую ошибку (забудут запятую или скобку).
- «Сгаллюцинируют» несуществующие поля.
- Искажают данные, пытаясь их «улучшить».

**Решение:** Перенести генерацию строгого JSON на сторону Python (`widget_renderer.py`, где это делается детерминировано и без ошибок), а LLM оставили роль **куратора/выборщика**

---

### Два типа виджетов

#### Тип А: Системные виджеты (`BarChart`, `LineChart`, `ScatterChart` и тд)

Эти виджеты рендерятся из данных инструментов, которые модель вызывала. LLM не генерирует JSON - она вставляет плейсхолдер в Final Answer:

```
<!-- SYSTEM_WIDGET: имя_intent | FILTER: {"key": "value"} | TITLE: Заголовок -->
```

Python-агент парсит плейсхолдер, находит в реестре соответствие intent -> tool_name, берёт данные из `data_cache`, фильтрует их по `FILTER`, и вызывает `render_widget()` из `widget_renderer.py`, который детерминированно строит ECharts-конфиг.

#### Тип Б: Модельные виджеты (`action_card` и др)

Эти виджеты требуют LLM-генерации содержимого (message, button и тд). LLM генерирует полный JSON-блок в Final Answer, обёрнутый в ```json ... ```. Python-агент прогоняет его через `render_widget()` для валидации и приведения к стандартному envelope

---

### ЖЦ виджета

#### Этап А: Декларация в SKILL.md

Автор плагина в YAML-фронтматтере скилла объявляет, какие типы визуализаций поддерживаются в контексте именно этого скилла, зачем они нужны и как сопоставить данные источников в ИС* с осями диаграммы:

```yaml
available_widgets:
  - type: ScatterChart
    intents:
      - name: risk_matrix_priority
        tool: get_risk_register
        description: "Матрица рисков: ось X — вероятность, ось Y — воздействие. Используй для ответа на вопрос «КАКИЕ риски наиболее опасны»."
        config:
          x: probability_numeric
          y: impact_numeric
          color_field: risk_score
          label: risk_id
          size_field: estimated_cost_impact
          thresholds:
            - {value: 9, color: "#ee6666", label: "Критический"}
            - {value: 6, color: "#fac858", label: "Высокий"}
            - {value: 0, color: "#91cc75", label: "Средний/Низкий"}
          x_axis: {min: 0, max: 4, name: "Вероятность"}
          y_axis: {min: 0, max: 4, name: "Воздействие"}

  - type: action_card
    intents:
      - name: mitigation_plan_action
        description: "Интерактивная карточка для предложения плана реагирования на риск."
        config: {}
```

*) - Внимание. Здесь паллиатив - фактически виджет иллюстрирует тул. Не очень сложно для видежтов сделать свои источники данных, но это решение надо взвешивать, тк описание виджета тогда станет альтернативным описанием уникального инструмента. Текущее решение оставлено осознанно.

**Смысл:** `config` выступает мостом между универсальным генератором (`widget_renderer.py`) и специфической структурой данных источника в ИС. В одном скилле можно задекларировать несколько intents одного типа, но разного назначения.

Виджеты `_type: action_card` объявляются как «особые» (`SPECIAL_WIDGET_TYPES`), они всегда force-included в `concentrator`-режиме, чтобы LLM могла сгенерировать план реагирования, если найдёт активные риски с высоким воздействием

#### Этап Б: Каталог в промпте

При построении промпта (как в `concentrator`, так и в `stateless`) агент собирает каталог виджетов из всех выбранных скиллов:

- В `concentrator`-режиме, концентратор сжимает методички, но `config` виджетов возвращается вербатим из YAML (LLM не может его исказить).
- В `stateless`-режиме, каталог виджетов с дословным `config` подаётся в промпт целиком. Модель видит его на каждой итерации, чтобы понимать, какие данные нужно собрать для какого виджета

#### Этап В: Выбор и плейсхолдер (LLM)

LLM получает в промпте каталог виджетов и Final Answer формирует так:

1. **Анализ:** LLM читает поле `description` каждого intent и сопоставляет его с намерением пользователя.
2. **Выбор:** Если запрос пользователя совпадает с описанием intent-а, LLM вставляет плейсхолдер:
   ```
   <!-- SYSTEM_WIDGET: risk_matrix_priority | FILTER: {} | TITLE: Матрица рисков проекта А -->
   ```
3. **Контекстуализация:** LLM обязана заменить техническое имя intent в `TITLE` на осмысленный, контекстно-зависимый заголовок.
4. **Гарантия целостности:** LLM не должнаизменять `config`, он берётся вербатим из YAML. LLM только указывает `intent` и `FILTER`.
5. **Вывод:** LLM вставляет плейсхолдеры в `Final Answer`, которые затем парсятся Python-агентом.

Для `action_card` LLM генерирует полный JSON-блок (с `widget_type`, `intent`, `title`, `message`, `button`), обёрнутый в ```json ... ```. Поля `message` и `button` и другие, заполняются на основе данных из Observation (фактических данных).

#### Этап Г: Рендеринг

Python-агент в `_render_system_widgets` (для плейсхолдеров) и `expand_widget_descriptors` (для JSON-блоков):

1. Парсит плейсхолдер -> извлекает `intent`, `FILTER`, `TITLE`.
2. Находит в реестре соответствие `intent -> tool_name` (в каталоге скилла).
3. Берёт данные из `data_cache[tool_name]` - это результат вызова инструмента.
4. Применяет спец-правила фильтрации (например, для прогнозов EAC отбрасывает строки `scenario="actual"`).
5. Применяет `FILTER` из плейсхолдера.
6. Вызывает `render_widget(descriptor)` из `widget_renderer.py`, передавая `widget_type`, `config`, `data_rows`, `title`.
7. `widget_renderer.py` детерминированно строит ECharts-конфиг (`option`) и возвращает envelope `{widget_type: "echarts", chart_type, intent, title, option}`.
8. Python-агент заменяет плейсхолдер на ```json ... ``` блок с envelope.

UI-клиент парсит эти блоки и отрисовывает ECharts-диаграммы или action_card

### Системная защита виджетов (stateless-режим)

В stateless-режиме перед возвратом Final Answer выполняется **принудительная проверка** (`_check_widgets_data_presence`):

1. Парсятся все плейсхолдеры `<!-- SYSTEM_WIDGET -->` в Final Answer.
2. Для каждого intent-а находится питающий `tool_name` в реестре.
3. Проверяется, что `tool_name` есть в `data_cache` (т.е. инструмент был вызван).
4. Если хотя бы одного нет, Final Answer **отклоняется**, и в накопленный блок данных добавляется уведомление:
   > «СИСТЕМНОЕ УВЕДОМЛЕНИЕ: ваш прошлый Final Answer отклонён. В нём были виджеты, для которых нет данных, потому что питающие их инструменты не были вызваны. НЕОБХОДИМО ВЫЗВАТЬ: `calculate_evm`. Составьте план, где первый шаг — вызов недостающего инструмента, и выполните его. После получения данных сформируйте Final Answer заново.»

Цикл продолжается, модель на следующей итерации видит это уведомление и должна вызвать недостающий инструмент. Это решает проблему «срезания углов»: модель не может вставить виджет `evm_curves` (требующий `calculate_evm`), если `calculate_evm` не был вызван.

Во вспомогательном `_generate_final_answer_stateless` (когда модель уже решила «одношаговый Final Answer», но не оформила сам ответ), проблемные плейсхолдеры заменяются на комментарий `<!-- Widget 'X' опущен: нет данных от инструмента 'Y' -->`, чтобы в UI не вылезало «No cached data for tool»
