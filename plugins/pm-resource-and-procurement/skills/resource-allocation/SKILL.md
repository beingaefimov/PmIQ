# Resource Allocation

Реализует процессы **Acquire Resources**, **Develop Team** и **Control Resources** (PMBOK 8),
методы CCPM в части ресурсных буферов и принцип теории ограничений (TOC):
перегруженный ресурс на критическом пути — это ограничение всей системы.

## Методологический контекст

### Пороги загрузки

| Уровень | Диапазон | Интерпретация |
|---|---|---|
| Критическая перегрузка | > 120% | Немедленное вмешательство |
| Перегрузка | 101–120% | Требует выравнивания |
| Рабочий оптимум | 70–100% | Норма |
| Рекомендуемый лимит | ≤ 80% | Для специалистов на нескольких проектах |
| Недогрузка | < 60% | Возможность перераспределения |

### Методы выравнивания

| Метод | Влияние на сроки | Влияние на стоимость | Когда |
|---|---|---|---|
| Resource Smoothing | Нет | Нет | Есть float у задач |
| Resource Leveling | Может увеличить | Нет | Float исчерпан |
| Crashing | Нет | Увеличивает | Срок критичен, бюджет есть |
| Fast Tracking | Риск переделок | Нет | Последовательные → параллельные |

### Матрица критичности перегрузки

| На критическом пути | Длительность | Уникальный навык | Приоритет |
|---|---|---|---|
| Да | > 3 нед | Да | 🔴 КРИТИЧЕСКИЙ — риск срыва сроков и "точка отказа" |
| Да | > 3 нед | Нет | 🟠 ВЫСОКИЙ — риск срыва сроков |
| Да | ≤ 3 нед | Да | 🟠 ВЫСОКИЙ — риск "точки отказа" |
| Да | ≤ 3 нед | Нет | 🟡 СРЕДНИЙ — мониторинг |
| Нет | > 3 нед | Да | 🟡 СРЕДНИЙ — риск потери компетенции |
| Нет | > 3 нед | Нет | 🟢 НИЗКИЙ — плановое выравнивание |
| Нет | ≤ 3 нед | Любой | 🟢 НИЗКИЙ — наблюдение |

## Инструкции для агента

### Шаг 1 — Получи данные загрузки
Вызови `get_resource_histogram` с фильтром по `project_id` (если известен, иначе - по всем проектам, то есть без значения фильтра).

### Шаг 2 — Оцени критичность перегрузки
Для каждого перегруженного ресурса (`allocation_pct > 100`) проверь по полям:
- `on_critical_path = true` → ресурс на критическом пути, его перегрузка напрямую угрожает срокам.
- `overload_duration_weeks > 3` → длительная перегрузка, риск выгорания.
- `unique_critical_skill = true` → риск "единственной точки отказа", нельзя просто перераспределить.

Используй **Матрицу критичности перегрузки** выше для определения приоритета.

### Шаг 3 — Выбери метод выравнивания
- Float есть (ресурс НЕ на критическом пути) → Resource Smoothing.
- Float исчерпан (ресурс на критическом пути) → Leveling (сдвиг сроков) или Crashing (бюджет).
- Уникальный навык + перегрузка → найм/аутсорсинг, нельзя просто перераспределить.
- Системная перегрузка нескольких ролей → пересмотр плана + выполни правила `skill-gap-analyzer`.

### Шаг 4 — Выбери intent для виджета
- "Кто перегружен?" → `overload_by_person`
- "Какая роль — узкое место?" → `load_by_role`
- "Есть ли свободные ресурсы для новой задачи?" → `availability_gap`
- "Кто на критическом пути и насколько перегружен?" → `critical_path_resources`

### Шаг 5 — Флажь риск выгорания
Загрузка > 100% в течение 3+ недель (`overload_duration_weeks >= 3`) → риск выгорания,
выполни правила `team-health-monitor`.
