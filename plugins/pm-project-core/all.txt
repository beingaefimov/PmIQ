# Собирает все файлы по папкам папки в один
# python3 merge.py

import os
with open('all.txt','w',encoding='utf-8') as out:
    for r,d,files in os.walk('.'):
        for f in files:
            p=os.path.join(r,f)
            if os.path.abspath(p)==os.path.abspath('all.txt'): continue
            out.write(f'\n===== {p} =====\n')
            try: out.write(open(p,'r',encoding='utf-8',errors='ignore').read())
            except: pass

Core project artifacts, schedules, WBS, and milestones for predictive, adaptive, and hybrid methodologies.

## Installation
```bash
copilot plugin install ./plugins/pm-project-core
```

*Note: Restart your Copilot CLI session after installation.*

# Project Identifier

## Инструкция
Этот скилл используется для разрешения нечётких упоминаний проектов в запросах пользователей.

### Когда вызывать ПЕРВЫМ:
- Если пользователь упоминает название проекта (даже приблизительно)
- Если пользователь спрашивает "какой проект..." или "найди проект..."
- Если в запросе есть контекст, связанный с конкретным проектом

### Как использовать результат:
После получения `project_id` из этого инструмента, используйте его как фильтр в последующих вызовах других инструментов.

### Примеры:
- Пользователь: "Покажи риски в Строительстве Площадки А"
  → Вызовите `identify_project(query="Строительство Площадки А")` → получите `project_id="Place_A_construction"`
  → Затем вызовите, например `get_some_register(project_name="Place_A_construction")`

- Пользователь: "Какие есть перегрузки в проекте по внедрению?"
  → Вызовите `identify_project(query="проект по внедрению")` → получите список кандидатов
  → Если один кандидат — используйте его, если несколько — уточните у пользователя

# Document Tracker

Фокусируемся на управлении конфигурацией и статусами документов.

## Инструкции для агента
1.  **Идентифицируйте проект:** Всегда начинайте с `project-identifier`, чтобы получить точное `project_name`.
2.  **Получите статус:** Используйте `get_document_approval_status` для проверки текущего статуса документа (`In Progress`, `Under Review`, `Approved`, `Rejected`).
3.  **Анализируйте замечания:** Если статус `Under Review` или `Rejected`, запросите детали из `comments` и определите количество критических замечаний.
4.  **Свяжите с расписанием:** Для строительных/инженерных проектов свяжите статус чертежей с вехами через `get_milestone_status`. Задержка согласования часто блокирует начало работ.
5.  **Предложите действия:** На основе статуса предложите следующие шаги (например, "Назначить ответственных за устранение замечаний" или "Запланировать встречу с экспертом").

# PM Schedule Tracker

Реализует домен производительности **Schedule** (PMBOK 8) и принцип системного мышления.

## Методологический контекст

| Методология | Артефакт расписания | Единица прогресса | Источник данных |
|---|---|---|---|
| Предиктивная | Диаграмма Ганта, базовый план | % завершения задачи | MS Project, Primavera P6 |
| Адаптивная (Scrum) | Спринт-бэклог, burn-down | Story Points, velocity | Jira, Azure DevOps |
| Адаптивная (Kanban) | CFD | Cycle time, throughput | Jira |
| Гибридная | Вехи + спринты | % фазы + velocity | MS Project + Jira |
| CCPM | Сетевой график с буферами | % потребления буфера | Специализированные инструменты |

## Инструкции для агента

### Шаг 1 — Определи методологию и контекст
Предиктивный, адаптивный или гибридный? Есть ли утверждённый базовый план?
Горизонт вопроса: оперативный (текущая неделя) или стратегический (дата завершения)?

### Шаг 2 — Получи данные расписания
- `get_project_schedule` → `completion_pct`, `critical_path_variance_days`, `end_date`.
- `get_wbs_structure` → незавершённые пакеты работ и их статус.
- `get_milestone_status` → вехи со статусом `At Risk` или `Delayed`.

### Шаг 3 — Рассчитай прогноз завершения
Если есть отклонение от базового плана:
- **SV** = EV − PV. Отрицательный SV = отставание (используй кросс-данные из EVM).
- **Прогнозная дата** = Плановая дата + `critical_path_variance_days`.
- CCPM: зона внимания при потреблении буфера > 50% при < 50% завершения проекта.

### Шаг 4 — Проверь связанные данные
Всегда проверяй `get_document_approval_status` для вех: незакрытые замечания по ТЗ
часто блокируют начало следующей фазы. Если отклонение > 5 дней — предложи
анализ через `pm-risk-and-change`.

### Шаг 5 — Выбери intent для виджета
- Вопрос "какие фазы отстают?" → `schedule_variance_by_phase`
- Вопрос "статус вех проекта?" → `milestone_status_overview`
- Вопрос "насколько сделана работа?" → `completion_progress`

### Шаг 6 — Сформулируй вывод
1. **Что** задержано (конкретная задача/веха).
2. **На сколько** (дни, % от SPI).
3. **Почему** (если данные есть).
4. **Что это означает** для итоговой даты проекта.
5. **Рекомендуемое действие** (crash, fast-track, CR через CCB).

## Required Tools
- `get_project_schedule`
- `get_wbs_structure`
- `get_milestone_status`
- `get_document_approval_status`
