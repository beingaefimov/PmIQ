# Team Health Monitor

Реализует домен производительности **Team** (PMBOK 8) и принцип
**Создание среды для успеха команды**. Опирается на модели командного развития
(Тукман: Forming → Storming → Norming → Performing → Adjourning),
психологической безопасности (Эдмондсон) и метрики Accelerate (DORA).

## Методологический контекст

### Признаки выгорания (ранние индикаторы)

| Сигнал | Источник данных | Порог внимания |
|---|---|---|
| Постоянные сверхурочные | Таймшиты, метаданные коммитов | > 50ч/неделю × 2+ недели |
| Рост числа дефектов | Jira, QMS | Рост defect rate > 25% за период |
| Снижение активности коммуникаций | MS Graph API метаданные | Падение > 30% от baseline |
| Снижение velocity | Jira sprint metrics | Падение > 20% за 2 спринта |
| Рост time-to-review в PR | GitHub / Azure DevOps | Рост > 50% от baseline |

### Принцип деликатности
Этот скилл работает только с **агрегированными и анонимизированными** данными команды.
Никогда не публикуй личные показатели отдельных участников публично.
При обнаружении индивидуальных рисков — рекомендуй PM провести приватный 1-on-1.

## Инструкции для агента

### Шаг 1 — Получи данные о здоровье команды
Вызови `get_feedback_summary`. При доступности — `analyze_communication_patterns`.
Получи данные загрузки - вызови `get_resource_histogram`.

### Шаг 2 — Классифицируй сигналы
Для каждой команды:
- `morale_score` < 0.5 → критично, требует немедленных действий PM.
- `burnout_indicators` содержит записи → детализируй по типам сигналов.
- `collaboration_score` < 0.6 → информационный вакуум, риск для качества.

### Шаг 3 — Свяжи с метриками проекта
Низкий morale_score + рост defect_rate → системная связь. Флажь это как риск в
`pm-risk-and-change`. Низкая коллаборация между Dev и QA → источник дефектов,
рекомендуй увеличить частоту синхронизаций.

### Шаг 4 — Выбери intent для виджета
- "В какой команде проблемы с климатом?" → `morale_by_team`
- "Из чего складывается риск выгорания?" → `burnout_signals_breakdown`
- "Налажено ли взаимодействие?" → `collaboration_health`

### Шаг 5 — Сформулируй рекомендации деликатно
Не: "У Иванова показатели хуже всех." 
Да: "В команде разработки 2 участника работают > 50 часов в неделю. Рекомендую PM
провести 1-on-1 встречи и рассмотреть перераспределение задач."

## Используемые MCP инструменты

- `analyze_communication_patterns`
