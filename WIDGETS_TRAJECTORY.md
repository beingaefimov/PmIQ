## 2. ТЕКУЩАЯ ЗАДАЧА: СОЗДАНИЕ UI-ВИДЖЕТОВ
Мы переходим от текстовых ответов агента к генерации интерактивных UI-виджетов, которые рендерятся прямо в окне чата (например, GitHub Copilot Chat, VS Code или Teams).

**Архитектурное ограничение (ВАЖНО!):**
Локальная LLM (7B-8B параметры) **НЕ МОЖЕТ** и **НЕ ДОЛЖНА** генерировать сложный JSON для виджетов (например, Microsoft Adaptive Cards или OpenAI Apps SDK) самостоятельно. Она гарантированно допустит синтаксическую ошибку или галлюцинацию в схеме.

**Правильный паттерн (уже заложенный в скилл `pm-dashboard-widget-builder`):**
1. LLM получает сырые табличные данные из MCP-инструмента (наш CSV-мокер).
2. LLM понимает, что данные лучше показать визуально, и вызывает специальный MCP-инструмент (например, `generate_widget_schema`), передавая ему *простые* параметры (тип виджета и сырые данные).
3. **MCP-сервер** (детерминированный Python-код) берет эти данные и оборачивает их в *строго валидный* JSON-формат виджета.
4. MCP-сервер возвращает этот JSON агенту, а агент передает его в чат для рендеринга.

## 4. ПОСЛЕДНЕЕ ОБЪЯСНЕНИЕ АРХИТЕКТУРЫ (ДЛЯ СПРАВКИ)
*Ниже приведено точное объяснение того, что такое UI-виджеты в нашем контексте, чтобы ты понимал, почему мы делаем именно так:*

"В контексте AI-ассистентов 'создание UI-виджетов' означает создание интерактивных визуальных элементов (графиков, диаграмм, кнопок), которые рендерятся прямо внутри окна текстового чата с ИИ. 

Языковая модель (LLM) не умеет рисовать графики или генерировать картинки. Она умеет генерировать только текст. Поэтому процесс выглядит так:
1. Пользователь задает вопрос в чате.
2. LLM понимает, что нужны визуальные данные, и вызывает специальный MCP-инструмент (например, `generate_widget_schema`), передавая ему сырые данные (например, из нашего CSV).
3. MCP-сервер (наш Python-код) берет эти данные и оборачивает их в строго типизированный JSON-формат, который понимает клиент чата (чаще всего это стандарт Microsoft Adaptive Cards или OpenAI Apps SDK).
4. Клиент чата (например, VS Code или Teams) получает этот JSON, парсит его и сам отрисовывает красивый график или кнопку прямо в интерфейсе.

Примеры для PM IQ:
- **Гистограмма ресурсов:** Столбчатая диаграмма, где загрузка >100% окрашена в красный.
- **Тепловая карта рисков:** Матрица Probability vs Impact, где точки размещаются в ячейках.
- **Интерактивные кнопки:** Карточка с кнопкой '[ ✉️ Отправить напоминание ]' или '[ 📝 Создать Change Request ]'.

Главное правило: LLM только *запрашивает* виджет, передавая простые параметры. Детерминированный MCP-сервер *генерирует* безупречный JSON схемы. Это предотвращает галлюцинации и синтаксические ошибки в сложных структурах виджетов.

Нам нужно будет изменить наш unified_mock_server.py (или написать реальные серверы).
Вместо того чтобы просто возвращать сырые строки из CSV:

{
  "data": [{"resource_name": "Alexeev A.", "allocation_pct": 120}]
}

Мы добавим инструмент, который возвращает Adaptive Card JSON:

{
  "type": "AdaptiveCard",
  "version": "1.5",
  "body": [
    {
      "type": "TextBlock",
      "text": "Загрузка команды разработки",
      "weight": "Bolder"
    },
    {
      "type": "Chart",

      "data": [
        {"label": "Alexeev A.", "value": 120, "color": "#FF0000"},
        {"label": "Borisov B.", "value": 80, "color": "#00FF00"}
      ]
    }
  ]
}

(Примечание: это упрощенный пример, реальный Adaptive Card для графиков сложнее, но принцип тот же).
