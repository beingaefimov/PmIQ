# ТРАЕКТОРИЯ РАЗРАБОТКИ: PM IQ REAL API INTEGRATION

## 1. КОНТЕКСТ ПРОЕКТА (PM IQ Universal Edition)

## 3. СТАНДАРТНЫЙ ПЛАН ДЕЙСТВИЙ (BEST PRACTICES)
Когда я дам команду начать интеграцию, следуй этому пошаговому алгоритму:

### Шаг 1: Создание выделенных MCP-серверов (вместо unified_mock)
- Вместо одного мок-сервера, создаем отдельные Python-скрипты (или npm-пакеты) для доменов, например: `servers/jira_mcp_server.py`, `servers/sap_mcp_server.py`.
- Используем надежные HTTP-клиенты (например, `httpx` или `aiohttp` в Python) с поддержкой асинхронности.
- **ВАЖНО:** Никогда не хардкодить секреты (API keys, tokens). Все креды читаются строго из `env` переменных, переданных через `.mcp.json`.

### Шаг 2: Обновление `.mcp.json`
- Изменить `command` и `args` в каждом плагине, чтобы они указывали на новый реальный сервер (например, `python3 ../../servers/jira_mcp_server.py` или `npx -y @pm-iq/jira-mcp`).
- Актуализировать секцию `env`, указав реальные имена переменных (например, `JIRA_API_TOKEN`, `SAP_ODATA_URL`).

### Шаг 3: Адаптация `SKILL.md` (Критически важно!)
Инструкции для LLM должны измениться, чтобы учитывать реалии API:
- **Синтаксис запросов:** Если API требует специфичного языка (например, JQL для Jira или OData для SAP), `SKILL.md` должен содержать примеры: *"При вызове get_issues используй параметр jql_query, например: 'project = ERP AND status = Blocked'"*.
- **Пагинация:** Реальные API возвращают данные частями. `SKILL.md` должен инструктировать LLM: *"Если в Observation указано 'has_more: true', вызови инструмент снова с параметром 'page=2'"*.
- **Обработка ошибок:** Инструкция: *"Если инструмент вернул ошибку 404 или 401, не выдумывай данные. Сообщи пользователю о проблеме с доступом или неверном ID и попроси уточнить параметры"*.

### Шаг 4: Реализация устойчивости (Resilience) в коде сервера
- MCP-сервер должен перехватывать исключения (Timeout, ConnectionError, HTTP 4xx/5xx).
- Вместо падения сервера, он должен возвращать LLM структурированный JSON с ошибкой: `{"error": "Jira API returned 401: Invalid Token", "suggestion": "Check JIRA_API_TOKEN in environment"}`. Это позволяет LLM элегантно сообщить о проблеме пользователю.

### Шаг 5: Тестирование
- Написать интеграционные тесты (например, с использованием `pytest` и библиотеки `responses` или `pytest-httpx` для мокирования реальных HTTP-ответов), чтобы гарантировать, что сервер корректно преобразует ответ API в формат, ожидаемый LLM.

## 4. ПРИМЕР СЦЕНАРИЯ (Jira Integration)
1. **Пользователь:** "Какие задачи заблокированы в проекте ERP_Migration?"
2. **LLM (читая SKILL.md):** Понимает, что нужен `get_issue_log`. Формирует JQL: `project = "ERP_Migration" AND status = "Blocked"`.
3. **Агент:** Вызывает `get_issue_log` с аргументом `{"jql_query": "project = \"ERP_Migration\" AND status = \"Blocked\""}`.
4. **MCP Сервер (`jira_mcp_server.py`):** 
   - Берет `JIRA_API_URL` и `JIRA_API_TOKEN` из `env`.
   - Делает GET запрос к `/rest/api/3/search?jql=...`.
   - Получает JSON, извлекает только нужные поля (summary, assignee, key), чтобы не перегружать контекст LLM.
   - Возвращает чистый JSON агенту.
5. **LLM:** Формирует человеческий ответ на основе полученных реальных данных.

## 5. ПОСЛЕДНЕЕ ОБЪЯСНЕНИЕ АРХИТЕКТУРЫ (ДЛЯ СПРАВКИ)
*Почему мы делаем именно так, а не позволяем LLM работать с API напрямую:*

"Стандартный и нормальный способ интеграции LLM с корпоративными API — это использование паттерна **Tool Use / Function Calling через протокол MCP**. 

Мы **НИКОГДА** не даем LLM прямой доступ к HTTP-клиенту или сырым эндпоинтам API по трем причинам:
1. **Безопасность:** LLM может случайно отправить секретные токены в открытый чат или выполнить деструктивный запрос (DELETE), если ее промпт будет скомпрометирован (Prompt Injection). MCP-сервер выступает как безопасный шлюз (Gateway), который валидирует запросы.
2. **Трансформация данных (Data Shaping):** Ответы реальных API (например, SAP или Jira) огромны и содержат сотни нерелевантных полей. Если скормить это LLM, она "захлебнется" (превысит лимит токенов). MCP-сервер на стороне бэкенда делает "проекцию": берет огромный JSON от API и возвращает LLM только 3-4 ключевых поля, необходимых для ответа.
3. **Стабильность:** Реальные API меняются, падают, требуют сложной аутентификации (OAuth flow). MCP-сервер инкапсулирует эту сложность. Для LLM вызов инструмента всегда выглядит одинаково просто, независимо от того, что происходит "под капотом".

Таким образом, переход к реальным API означает замену *логики чтения CSV* внутри наших MCP-серверов на *логику HTTP-запросов с трансформацией ответа*, при этом интерфейс для LLM (название инструмента и его аргументы) остается максимально стабильным."
